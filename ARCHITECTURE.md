# Архитектура сервиса

## Выбор архитектуры

Для учебного проекта выбран монолитный сервис. В одном приложении находятся
загрузка модели, валидация входных данных и HTTP API. Такой вариант проще
запустить локально и в Docker, а объём логики небольшой.

Микросервисная архитектура была бы полезна при большем числе моделей, отдельных
командах разработки и высоких требованиях к масштабированию. Для текущего
кейса отдельные сервисы добавили бы больше сложности чем пользы.

## RabbitMQ

RabbitMQ можно добавить при росте нагрузки. Очередь пригодилась бы для:

- асинхронной обработки батч-предсказаний;
- записи событий скоринга без задержки ответа клиенту;
- передачи логов или заявок в отдельный сервис мониторинга.

В текущей версии брокер описан концептуально, реализация не добавлена.

## Логирование и мониторинг

API пишет запросы и ответы в `logs/api_requests.jsonl`. Формат одной строки:

```json
{"request": {"features": {"limit_bal": 120000}}, "response": {"prediction": 1}}
```

В production такие логи можно собирать через ELK-стек или другой log collector.
По ним можно смотреть количество запросов, ошибки валидации, распределение
вероятностей дефолта и долю положительных прогнозов.

## DVC и MLflow

DVC нужен для контроля версий данных и обученных моделей. Он полезен если
датасет меняется и нужно понимать, на какой версии данных обучалась модель.

MLflow нужен для отслеживания экспериментов. В нём можно хранить параметры
моделей, метрики, артефакты и сравнивать версии `v1` и `v2`.

## ONNX-ML

Модель scikit-learn можно перевести в ONNX-ML через `skl2onnx`. Это может быть
полезно для ускорения инференса и запуска модели в средах, где Python не нужен.
Для текущего сервиса оставлен формат `joblib`, так как он проще для учебной
работы.

## uWSGI + NGINX

В production Flask обычно не запускают встроенным сервером. uWSGI или gunicorn
запускает Python-приложение несколькими воркерами, а NGINX принимает внешний
трафик, проксирует запросы, отдаёт статические файлы и может ограничивать
нагрузку.
